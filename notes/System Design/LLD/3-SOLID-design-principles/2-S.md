# Single Responsibility Principle (S)

## Definition

A class should have **one, and only one, reason to change**. Each class should have a single responsibility and should encapsulate only one reason to change.

---

## Key Points

- **Single Responsibility**: A class should do only one thing
- **One Reason to Change**: If a class has multiple reasons to change, it violates SRP
- **Cohesion**: Methods in a class should work together for a common purpose
- **Maintainability**: Easier to test, debug, and modify

---

## ❌ Before: Violating SRP

```python
class User:
    def __init__(self, name, email):
        self.name = name
        self.email = email

    # Responsibility 1: User Management
    def save_to_db(self):
        # Save user to database
        print(f"Saving {self.name} to database")

    # Responsibility 2: Email Notifications
    def send_email(self, message):
        # Send email to user
        print(f"Sending email to {self.email}: {message}")

    # Responsibility 3: Logging
    def log_user_activity(self, activity):
        # Log activity to file
        with open('log.txt', 'a') as f:
            f.write(f"{self.name}: {activity}\n")

    # Responsibility 4: Data Validation
    def validate_email(self):
        # Validate email format
        return '@' in self.email
```

**Problems:**

- 4 reasons to change: database schema, email service, logging format, validation logic
- Hard to test (requires database, file system, email service)
- Difficult to reuse validation or notification logic
- Classes become tightly coupled

---

## ✅ After: Following SRP

```python
# Responsibility 1: User Management
class User:
    def __init__(self, name, email):
        self.name = name
        self.email = email


# Responsibility 2: Database Persistence
class UserRepository:
    def save(self, user):
        print(f"Saving {user.name} to database")

    def get_by_email(self, email):
        print(f"Fetching user with email {email}")


# Responsibility 3: Email Notifications
class EmailService:
    def send(self, email, message):
        print(f"Sending email to {email}: {message}")


# Responsibility 4: Logging
class Logger:
    def log(self, message):
        with open('log.txt', 'a') as f:
            f.write(f"{message}\n")


# Responsibility 5: Validation
class EmailValidator:
    @staticmethod
    def is_valid(email):
        return '@' in email and '.' in email.split('@')[1]
```

**Benefits:**

- ✅ Each class has one reason to change
- ✅ Easy to test (can mock dependencies)
- ✅ Highly reusable
- ✅ Loose coupling
- ✅ Clear responsibilities
- ✅ Easy to maintain and extend

---

## Usage Example

```python
# Create components
user = User("John Doe", "john@example.com")
repo = UserRepository()
email_service = EmailService()
logger = Logger()
validator = EmailValidator()

# Validate email
if validator.is_valid(user.email):
    # Save user
    repo.save(user)
    # Send welcome email
    email_service.send(user.email, "Welcome!")
    # Log activity
    logger.log(f"User {user.name} created successfully")
else:
    logger.log(f"Invalid email: {user.email}")
```

## Real-World Example: Order Processing

### ❌ Before (SRP Violation)

```python
class Order:
    def __init__(self, order_id, items):
        self.order_id = order_id
        self.items = items

    def calculate_total(self):
        return sum(item.price for item in self.items)

    def save_to_db(self):
        print(f"Saving order {self.order_id}")

    def send_confirmation_email(self, customer_email):
        print(f"Sending confirmation to {customer_email}")

    def generate_invoice_pdf(self):
        print("Generating PDF invoice")

    def process_payment(self, card_details):
        print("Processing payment...")
```

### ✅ After (Following SRP)

```python
class Order:
    def __init__(self, order_id, items):
        self.order_id = order_id
        self.items = items

    def calculate_total(self):
        return sum(item.price for item in self.items)


class OrderRepository:
    def save(self, order):
        print(f"Saving order {order.order_id}")


class OrderNotificationService:
    def send_confirmation(self, email, order):
        print(f"Sending confirmation to {email}")


class InvoiceGenerator:
    def generate_pdf(self, order):
        print("Generating PDF invoice")


class PaymentProcessor:
    def process(self, order, card_details):
        print("Processing payment...")
```

---

## How to Identify SRP Violations

1. **Multiple reasons to change**: If you can list more than one reason to modify the class
2. **Multiple methods with different purposes**: Class does different things
3. **Difficult testing**: Need to mock too many dependencies
4. **Confused naming**: Class names become vague ("Handler", "Manager", "Processor")
5. **Large files**: Class is too long and does too much

---

## Summary

| Aspect            | Before (Violated) | After (SRP) |
| ----------------- | ----------------- | ----------- |
| Responsibilities  | Multiple          | One         |
| Reasons to Change | Many              | One         |
| Testability       | Hard              | Easy        |
| Reusability       | Poor              | High        |
| Coupling          | Tight             | Loose       |
| Maintainability   | Difficult         | Easy        |
| Code Size         | Large             | Focused     |
