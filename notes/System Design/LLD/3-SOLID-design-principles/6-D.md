# D — Dependency Inversion Principle (DIP)

## Definition (Interview-ready)

High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details; details should depend on abstractions.

Short line: "Depend on interfaces/abstractions, not concrete implementations."

## Why it matters

- Decouples high-level policy from low-level implementation details.
- Improves testability (can inject mocks/stubs).
- Makes swapping implementations (e.g., DB, external services) low-risk.

## Common mistakes / smell

- High-level code directly instantiates concrete classes (new Stripe(), new MySQLRepo()).
- Lots of `if`/`switch` statements to select concrete implementations.

## Bad example — high-level depends on low-level

```python
class StripeProcessor:
	def charge(self, amount):
		print("Charging via Stripe")


class CheckoutService:
	def __init__(self):
		# direct dependency on a concrete class -> rigid
		self.processor = StripeProcessor()

	def pay(self, amount):
		self.processor.charge(amount)
```

Problems: `CheckoutService` is hard to test and change. To use PayPal, modify `CheckoutService`.

## Good example — depend on abstraction + inject

```python
from abc import ABC, abstractmethod

class PaymentProcessor(ABC):
	@abstractmethod
	def charge(self, amount: float) -> bool:
		pass


class StripeProcessor(PaymentProcessor):
	def charge(self, amount: float) -> bool:
		print("Stripe charged", amount)
		return True


class PayPalProcessor(PaymentProcessor):
	def charge(self, amount: float) -> bool:
		print("PayPal charged", amount)
		return True


class CheckoutService:
	def __init__(self, processor: PaymentProcessor):
		self.processor = processor  # injected abstraction

	def pay(self, amount: float):
		return self.processor.charge(amount)


# Usage: inject concrete implementation at composition root
checkout = CheckoutService(StripeProcessor())
checkout.pay(100)
```

Benefits:

- `CheckoutService` depends only on `PaymentProcessor` abstraction.
- Can easily swap or mock processors in tests.

## Relation to IoC and DI

- DIP is the design principle: depend on abstractions.
- Inversion of Control (IoC) is a broader pattern where control is inverted (frameworks, containers).
- Dependency Injection (DI) is a technique to satisfy dependencies (constructor, setter, or service locator).

## When to apply / trade-offs

- Use DIP when high-level modules have changing low-level needs or to enable testing.
- DI introduces wiring (composition root) which should be managed carefully.

## Interview Qs — concise answers

- Q: "What's DIP?" — "High-level modules depend on abstractions, not concretes."
- Q: "How to implement?" — "Use interfaces/abstract classes and inject concrete implementations at runtime."
- Q: "Why not always use DI frameworks?" — "For small apps, simple constructor injection is enough; frameworks add complexity."

## Quick checklist

- Do high-level modules instantiate concrete dependencies directly?
- Can you swap an implementation without changing the high-level module?
- Is there a clear composition/wiring place (composition root)?
