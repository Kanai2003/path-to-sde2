# I — Interface Segregation Principle (ISP)

## Definition (Interview-ready)

Clients should not be forced to depend on interfaces they do not use. Prefer many small, specific interfaces over one large, general-purpose interface.

Short line to say in interviews: "Split fat interfaces so each client only sees what it needs."

## Why it matters

- Reduces needless coupling between clients and unwanted methods.
- Makes implementations simpler, easier to test and evolve.
- Prevents change ripples when unrelated methods change.

## Common smell / violation

- A single interface/class exposes many methods but concrete implementers only need a subset.
- Client code has unused method implementations or raises NotImplementedError for methods it doesn't support.

## Concrete (bad) example — fat interface

```python
from abc import ABC, abstractmethod

class Worker(ABC):
	@abstractmethod
	def work(self):
		pass

	@abstractmethod
	def eat(self):
		pass


class RobotWorker(Worker):
	def work(self):
		print("Robot working")

	# Robot doesn't eat — forced to implement or raise
	def eat(self):
		raise NotImplementedError("Robots don't eat")
```

Problems: `RobotWorker` is forced to carry `eat()` which is irrelevant — client must guard or deal with exceptions.

## Good design — split interfaces

```python
from abc import ABC, abstractmethod

class Workable(ABC):
	@abstractmethod
	def work(self):
		pass


class Feedable(ABC):
	@abstractmethod
	def eat(self):
		pass


class HumanWorker(Workable, Feedable):
	def work(self):
		print("Human working")

	def eat(self):
		print("Human eating")


class RobotWorker(Workable):
	def work(self):
		print("Robot working")


def run_shift(worker: Workable):
	worker.work()
```

Benefits:

- No NotImplementedError hacks.
- Clients depend only on the small interface they need (`Workable`).
- Easier mocking and testing.

## When to apply / trade-offs

- Apply when an interface has clear groups of unrelated methods.
- Avoid premature fragmentation: too many tiny interfaces may increase complexity.
- Aim for pragmatic splits: group methods by client role or capability.

## Interview Qs — concise answers

- Q: "What is ISP?" — "Clients shouldn't depend on methods they don't use; split fat interfaces."
- Q: "How to fix a violation?" — "Extract smaller interfaces and make clients depend on the minimal one."
- Q: "Why not just keep one interface?" — "It creates coupling, forces irrelevant code, and breaks substitutability."

## Quick checklist for design review

- Does any implementing class raise NotImplementedError for interface methods?
- Do clients call a large interface but only use a subset of methods?
- Can you extract a smaller role-specific interface?
