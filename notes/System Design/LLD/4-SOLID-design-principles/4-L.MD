# L - Liskov Substitution Principle (LSP)

If class B is a subtype of class A, then we should be able to use B wherever A is expected, without breaking behavior.

Simple line for interviews:
Subclasses must be replaceable for their base class.

---

## Why It Matters

- Prevents hidden runtime bugs.
- Keeps polymorphism safe.
- Makes APIs predictable.
- Reduces special-case checks like if type is X.

---

## Banking Analogy (Easy to Remember)

Base type: BankAccount

Expected behavior:

- deposit(amount) increases balance.
- withdraw(amount) either succeeds or fails by clear contract.
- balance never becomes invalid.

If SavingsAccount and CurrentAccount are BankAccount subtypes, both must honor the same contract so any service using BankAccount works unchanged.

---

## Bad Example (LSP Violation)

Problem: Child class changes expected behavior.

```python
class BankAccount:
	def __init__(self, balance: float = 0):
		self.balance = balance

	def withdraw(self, amount: float) -> None:
		if amount > self.balance:
			raise ValueError("Insufficient funds")
		self.balance -= amount


class FixedDepositAccount(BankAccount):
	# Fixed deposit should not allow withdrawal,
	# but overriding with runtime failure breaks substitutability.
	def withdraw(self, amount: float) -> None:
		raise NotImplementedError("Withdrawal not allowed")


def pay_bill(account: BankAccount, amount: float) -> None:
	account.withdraw(amount)


# Works for BankAccount, but breaks for FixedDepositAccount.
pay_bill(FixedDepositAccount(10000), 500)
```

Why this is wrong:

- Caller expects any BankAccount to support withdraw by base contract.
- Subtype throws unsupported behavior.
- Subtype is not safely replaceable.

---

## Good Design (LSP Compliant)

Split abstractions so each subtype supports only valid operations.

```python
from abc import ABC, abstractmethod


class Account(ABC):
	@abstractmethod
	def get_balance(self) -> float:
		pass


class WithdrawableAccount(Account):
	@abstractmethod
	def withdraw(self, amount: float) -> None:
		pass


class SavingsAccount(WithdrawableAccount):
	def __init__(self, balance: float = 0):
		self.balance = balance

	def withdraw(self, amount: float) -> None:
		if amount > self.balance:
			raise ValueError("Insufficient funds")
		self.balance -= amount

	def get_balance(self) -> float:
		return self.balance


class FixedDepositAccount(Account):
	def __init__(self, balance: float = 0):
		self.balance = balance

	def get_balance(self) -> float:
		return self.balance


def process_atm_withdrawal(account: WithdrawableAccount, amount: float) -> None:
	account.withdraw(amount)
```

Now contracts are clear:

- ATM withdrawal accepts only WithdrawableAccount.
- Fixed deposit is no longer forced into invalid behavior.

## Another Practical Banking Example

Base contract: Daily withdrawal limit check.

Wrong subtype behavior:

- CurrentAccount ignores limit and allows unlimited withdrawal.

Result:

- Fraud/risk module relying on base rules fails unexpectedly.

Correct approach:

- Keep limit logic in base contract, or
- Define separate account types/contracts for accounts with different rules.

## Detailed LSP Rules with Code Examples

### 1. Signature Rule — Method Arguments (Contravariance)

**Rule**: A subclass method should accept the same (or broader) argument types as the parent.

**Violation Example:**

```python
class Animal:
    def feed(self, food: object):
        print(f"Feeding with {food}")

class Dog(Animal):
    # Narrowed argument type — only accepts Dog-specific food
    def feed(self, food: DogFood):  # ❌ Too restrictive
        print(f"Dog eating {food}")

# Client code breaks
def care_for_animal(animal: Animal, food: object):
    animal.feed(food)  # works for Animal, might fail for Dog subtype

care_for_animal(Dog(), "generic food")  # TypeError: expects DogFood
```

**Correct Design:**

```python
class Animal:
    def feed(self, food: object):
        print(f"Feeding with {food}")

class Dog(Animal):
    def feed(self, food: object):  # ✅ Same or broader argument
        print(f"Dog eating {food}")

# Client code works universally
care_for_animal(Dog(), "generic food")  # OK
```

---

### 2. Signature Rule — Return Type (Covariance)

**Rule**: A subclass method can return the same type or a more specific (derived) type.

**Violation Example:**

```python
class Shape:
    def clone(self) -> Shape:
        return Shape()

class Circle(Shape):
    def clone(self) -> object:  # ❌ Returned type too broad
        return object()

# Client expects Shape, gets generic object
shape = Circle()
cloned = shape.clone()  # type is object, not Circle
```

**Correct Design:**

```python
class Shape:
    def clone(self) -> Shape:
        return Shape()

class Circle(Shape):
    def clone(self) -> Circle:  # ✅ More specific, still compatible
        return Circle()

# Client code works
circle = Circle()
cloned = circle.clone()  # type is Circle (or Shape is safe too)
```

### 3. Signature Rule — Exception Rule

**Rule**: A subclass method must throw the same or narrower exceptions (not broader) than the parent.

**Violation Example:**

```python
class PaymentGateway:
    def process(self, amount: float) -> bool:
        # Only raises ValueError for validation errors
        if amount <= 0:
            raise ValueError("Invalid amount")
        return True

class StripeGateway(PaymentGateway):
    def process(self, amount: float) -> bool:
        # ❌ Throws broader exception (unexpected by client)
        if amount <= 0:
            raise ValueError("Invalid amount")
        # Also throws Exception that base doesn't mention
        raise Exception("Stripe API is down")
        return True

# Client code assumes only ValueError
def checkout(gateway: PaymentGateway, amount: float):
    try:
        gateway.process(amount)
    except ValueError:
        print("Invalid amount")
    # Not catching Exception, so crashes

checkout(StripeGateway(), 100)  # Exception not caught → runtime error
```

**Correct Design:**

```python
class PaymentGateway:
    def process(self, amount: float) -> bool:
        if amount <= 0:
            raise ValueError("Invalid amount")
        return True

class StripeGateway(PaymentGateway):
    def process(self, amount: float) -> bool:
        if amount <= 0:
            raise ValueError("Invalid amount")  # ✅ Same exception
        return True

# Client code works
def checkout(gateway: PaymentGateway, amount: float):
    try:
        gateway.process(amount)
    except ValueError:
        print("Invalid amount")

checkout(StripeGateway(), 100)  # Safe
```

### 4. Property Rule (Invariants & History)

**Rule**: Subclasses must preserve class invariants. Properties that were non-null, non-negative, or immutable must remain so.

**Violation Example:**

```python
class BankAccount:
    def __init__(self, account_id: str, balance: float):
        self.account_id = account_id  # Invariant: account_id never None
        self.balance = balance  # Invariant: balance >= 0

    def get_account_id(self) -> str:
        return self.account_id

class TemporaryAccount(BankAccount):
    def close(self):
        self.account_id = None  # ❌ Breaks invariant
        self.balance = -100  # ❌ Breaks balance >= 0

# Client code assumes invariants
def record_transaction(account: BankAccount):
    account_id = account.get_account_id()
    id_length = len(account_id)  # None → TypeError
    
account = TemporaryAccount("ACC123", 500)
account.close()
record_transaction(account)  # Crashes
```

**Correct Design:**

```python
class BankAccount:
    def __init__(self, account_id: str, balance: float):
        self.account_id = account_id
        self.balance = balance

    def get_account_id(self) -> str:
        return self.account_id

class TemporaryAccount(BankAccount):
    def close(self):
        # ✅ Keep invariants: don't set to None or negative
        self.balance = 0  # Allow close with zero balance

# Client code works
account = TemporaryAccount("ACC123", 500)
account.close()
record_transaction(account)  # Safe
```

### 5. History Rule (Immutability Expectations)

**Rule**: If a property or contract assumes immutability or a fixed sequence, subclasses must honor it.

**Violation Example:**

```python
class Configuration:
    def __init__(self, api_key: str):
        self._api_key = api_key  # Implicit: api_key is read-only after init

    def get_api_key(self) -> str:
        return self._api_key

class MutableConfiguration(Configuration):
    def set_api_key(self, key: str):  # ❌ Breaks immutability
        self._api_key = key

# Client caches API key
def init_client(config: Configuration):
    key = config.get_api_key()  # Assume cached value is stable
    # ... later, key could change unexpectedly
    # Security issue: key swapped mid-session
    
config = MutableConfiguration("key-v1")
init_client(config)
config.set_api_key("key-v2")  # Client doesn't know; uses old key
```

**Correct Design:**

```python
class Configuration:
    def __init__(self, api_key: str):
        self._api_key = api_key

    def get_api_key(self) -> str:
        return self._api_key

class MutableConfiguration(Configuration):
    def __init__(self, api_key: str):
        super().__init__(api_key)
    
    # ✅ Don't mutate after creation; if needed, pass new config object

# Or, if mutation is needed, it's a separate, different type
class DynamicConfiguration:
    def __init__(self, api_key: str):
        self._api_key = api_key
    
    def set_api_key(self, key: str):
        self._api_key = key  # ✅ Not a subtype of Configuration
```

### 6. Pre-Condition Rule (Method Contracts)

**Rule**: A subclass method should not strengthen preconditions. It can only weaken them.

**Violation Example:**

```python
class PaymentProcessor:
    def charge(self, amount: float):
        # Precondition: amount must be > 0
        if amount <= 0:
            raise ValueError("Amount must be positive")
        print(f"Processing ${amount}")

class PremiumProcessor(PaymentProcessor):
    def charge(self, amount: float):
        # ❌ Strengthened precondition: now also requires amount > 100
        if amount <= 100:
            raise ValueError("Amount must be > 100")
        print(f"Processing ${amount}")

# Client code assumes base precondition
def pay_bill(processor: PaymentProcessor, amount: float):
    processor.charge(amount)  # Works for $50 with base, fails with Premium

pay_bill(PremiumProcessor(), 50)  # ValueError (too strict)
```

**Correct Design:**

```python
class PaymentProcessor:
    def charge(self, amount: float):
        if amount <= 0:
            raise ValueError("Amount must be positive")
        print(f"Processing ${amount}")

class PremiumProcessor(PaymentProcessor):
    def charge(self, amount: float):
        # ✅ Same or weaker precondition
        if amount <= 0:
            raise ValueError("Amount must be positive")
        print(f"Processing ${amount}")

pay_bill(PremiumProcessor(), 50)  # OK
```

### 7. Post-Condition Rule (Method Guarantees)

**Rule**: A subclass method must guarantee the same or stronger postconditions. It cannot weaken guarantees.

**Violation Example:**

```python
class OrderService:
    def place_order(self, item: str) -> str:
        # Postcondition: returns a valid order_id and order is in DB
        order_id = "ORD-123"
        print(f"Order {order_id} created")
        return order_id

class FastOrderService(OrderService):
    def place_order(self, item: str) -> str:
        # ❌ Weakened postcondition: returns ID but doesn't save to DB
        order_id = "ORD-456"
        print(f"Order {order_id} created (not saved yet)")
        return order_id

# Client code assumes order is persisted
def confirm_order(service: OrderService, item: str):
    order_id = service.place_order(item)
    # Query DB for order_id
    order = db.get_order(order_id)  # None for FastOrderService ❌
    return order

confirm_order(FastOrderService(), "widget")  # Order not found
```

**Correct Design:**

```python
class OrderService:
    def place_order(self, item: str) -> str:
        order_id = "ORD-123"
        db.save_order(order_id)
        return order_id

class FastOrderService(OrderService):
    def place_order(self, item: str) -> str:
        # ✅ Same or stronger postcondition: still saves, maybe optimized
        order_id = "ORD-456"
        db.save_order_async(order_id)  # Same guarantee, async
        return order_id

confirm_order(FastOrderService(), "widget")  # OK
```

## LSP Rules You Can Say in Interviews

1. **Do not strengthen preconditions** in child class.
   Example: Parent allows withdraw if amount > 0. Child should not require amount > 100.

2. **Do not weaken postconditions**.
   Example: Parent guarantees transaction is saved. Child must also guarantee it.

3. **Preserve invariants** (properties that must always be true).
   Example: balance must not go negative unless contract explicitly allows overdraft.

4. **Do not throw broader/unexpected exceptions** for normal parent use.
   Example: Parent throws ValueError for validation; child should not throw Exception.

5. **Method argument types** must be same or broader (contravariance).
   Example: Parent accepts object; child can accept object, not restrict to string.

6. **Return types** must be same or more specific (covariance).
   Example: Parent returns Shape; child can return Circle (subtype of Shape).

7. **Honor immutability contracts** (history rule).
   Example: If parent treats property as read-only after init, child must too.

---

## Smells That Indicate LSP Violation

- Subclass methods that say not supported.
- if isinstance checks everywhere before calling a method.
- Child class silently doing less or different than parent promise.
- Client code has special handling for specific subtype.

---

## LSP vs Other SOLID Principles

- SRP: one reason to change.
- OCP: extend behavior without modifying old code.
- LSP: extension must remain behaviorally compatible.

Quick way to connect:
OCP gives extension. LSP ensures extension is safe.

## Interview Q and A (Fast Revision)

Q: What is LSP?
A: Any subclass should be replaceable for its base class without breaking correctness.

Q: Classic violation example?
A: BankAccount with withdraw, and FixedDepositAccount throwing not supported for withdraw.

Q: How to fix?
A: Extract smaller, correct abstractions like WithdrawableAccount and use interface-based contracts.

Q: Real impact in production?
A: Runtime failures, unexpected behavior, security/risk rule bypass, and fragile code.

---

## Quick Checklist Before Finalizing Design

- Can every subtype be used where base type is expected?
- Are method contracts same or stronger in reliability?
- Any subtype throwing not supported for base methods?
- Any client forced to check concrete subtype?

If yes to last two, redesign abstraction.
