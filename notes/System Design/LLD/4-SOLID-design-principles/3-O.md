# O — Open/Closed Principle

> Classes/modules should be **open for extension, closed for modification**.

Add new features WITHOUT changing existing code. Extend behavior, don't rewrite it.

---

## Mantra

- **Open** = new functionality can be added
- **Closed** = existing code is stable, tested, released
- **Extension** = inheritance, composition, interfaces
- **Modification** = touching old code = risk

---

## Real World Analogy

Car factory makes cars. Car design is **closed** for modification (blueprint fixed).
But **open** for extension — add sunroof, leather seats, GPS. New features added without redesigning the whole car.

---

## Bad Example — Violates OCP

```python
class ReportGenerator:
    def generate_report(self, report_type: str):
        if report_type == "PDF":
            return self.generate_pdf()
        elif report_type == "CSV":
            return self.generate_csv()
        elif report_type == "EXCEL":
            return self.generate_excel()
        # Tomorrow: new type needed?
        # Modify existing class → risk of breaking old reports

    def generate_pdf(self): ...
    def generate_csv(self): ...
    def generate_excel(self): ...
```

**Problem:**

- Add new report type = modify existing class
- Each change = retest everything
- Tight coupling
- Violates **Single Responsibility** too

---

## Good Example — Follows OCP

```python
from abc import ABC, abstractmethod

class Report(ABC):
    @abstractmethod
    def generate(self) -> str:
        pass

class PDFReport(Report):
    def generate(self) -> str:
        return "PDF content..."

class CSVReport(Report):
    def generate(self) -> str:
        return "CSV content..."

class ExcelReport(Report):
    def generate(self) -> str:
        return "EXCEL content..."

# NEW report type? No need to modify ReportGenerator
class JSONReport(Report):
    def generate(self) -> str:
        return "JSON content..."

class ReportGenerator:
    def generate_report(self, report: Report) -> str:
        return report.generate()    # delegates to correct type

# Usage
pdf = PDFReport()
json_report = JSONReport()

gen = ReportGenerator()
gen.generate_report(pdf)        # works
gen.generate_report(json_report)  # works — no modification needed!
```

**Why it works:**

- Add new report type = new class only
- `ReportGenerator` never touched
- Existing reports guaranteed safe
- Open for extension (new Report subclass)
- Closed for modification (ReportGenerator stable)

## How to Achieve OCP

### 1. Abstraction (Interface/Abstract Class)

```python
class PaymentProcessor(ABC):
    @abstractmethod
    def process_payment(self, amount: float) -> bool:
        pass
```

### 2. Inheritance/Polymorphism

```python
class StripeProcessor(PaymentProcessor):
    def process_payment(self, amount: float) -> bool: ...

class PayPalProcessor(PaymentProcessor):
    def process_payment(self, amount: float) -> bool: ...
```

### 3. Composition/Dependency Injection

```python
class CheckoutService:
    def __init__(self, payment_processor: PaymentProcessor):
        self.processor = payment_processor  # inject dependency

    def checkout(self, amount: float):
        return self.processor.process_payment(amount)
```

New processor? Pass it in. No code change in `CheckoutService`.

## Interview Example — E-Commerce

**Scenario:** "Design order discount system."

**Bad approach:**

```python
class Order:
    def calculate_discount(self, customer_type: str):
        if customer_type == "VIP":
            return self.total * 0.20
        elif customer_type == "PREMIUM":
            return self.total * 0.10
        # Need bulk discount? Modify Order class
```

**Good approach:**

```python
class DiscountStrategy(ABC):
    @abstractmethod
    def calculate_discount(self, total: float) -> float:
        pass

class VIPDiscount(DiscountStrategy):
    def calculate_discount(self, total: float) -> float:
        return total * 0.20

class PremiumDiscount(DiscountStrategy):
    def calculate_discount(self, total: float) -> float:
        return total * 0.10

class BulkDiscount(DiscountStrategy):
    def calculate_discount(self, total: float) -> float:
        return total * 0.15 if total > 10000 else 0

class Order:
    def __init__(self, discount_strategy: DiscountStrategy):
        self.strategy = discount_strategy

    def get_final_price(self, total: float):
        discount = self.strategy.calculate_discount(total)
        return total - discount

# Usage — no changes to Order class
order1 = Order(VIPDiscount())
order2 = Order(BulkDiscount())
```

Tomorrow: loyalty discount? New class. `Order` never touched.

## OCP vs DRY vs Other Principles

| Principle | Focus                          | Tool                           |
| --------- | ------------------------------ | ------------------------------ |
| **OCP**   | Extension without modification | Interfaces, inheritance        |
| **SRP**   | One responsibility per class   | Small, focused classes         |
| **DRY**   | No code duplication            | Shared utilities, base classes |

OCP uses abstraction. If you violate SRP while following OCP, you still lose.

---

## Common Mistakes

❌ **Trying to predict ALL future changes** → bloated design, premature abstraction
✅ **Add abstraction when you see repetition or future extension need**

❌ **Deep class hierarchies** → hard to understand, hard to extend
✅ **Shallow hierarchies + composition** → cleaner, flexible

---

## Interview Talking Points

1. **What is OCP?** "Open for extension, closed for modification."
2. **Why matters?** "Lower risk. New features = new code, not touching old tested code."
3. **How achieve?** "Abstraction (interfaces), inheritance, polymorphism."
4. **Real example?** "Payment processors: base interface, each provider is new class."
5. **Trade-off?** "More abstraction = more files, can over-engineer. Balance needed."

---

## Quick Checklist

- [ ] Adding new feature requires modifying existing class?
- [ ] Yes → consider abstract base class or interface
- [ ] Classes depend on concrete types (not interfaces)?
- [ ] Yes → use dependency injection
- [ ] Same logic in multiple places?
- [ ] Yes → extract to shared base class or interface
